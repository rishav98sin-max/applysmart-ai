# Diagnostics — LLM Telemetry Layer

Per-call telemetry for the agent's LLM activity. Fully isolated from the main app.

## What it captures (per LLM call)

- `agent` — which module made the call (`cv_diff_tailor`, `reviewer`, `planner`, …)
- `provider` — `groq` / `gemini` / `groq_fallback` (when Gemini fell back)
- `model` — exact model id
- `total_tokens`, `prompt_chars`, `response_chars`, `duration_ms`
- `truncated` — heuristic flag (response near max_tokens or unclosed JSON)
- `error` — exception class + first 300 chars
- `job_id` — set by the orchestrator so all calls in one job group together
- `prompt` / `response` — trimmed to ~500 chars unless `DIAGNOSTICS_FULL_PAYLOAD=1`

Two backends, written in parallel:

1. **Local JSONL** — `diagnostics/runs/<timestamp>/traces.jsonl` (always on)
2. **Langfuse** — when `LANGFUSE_PUBLIC_KEY` + `LANGFUSE_SECRET_KEY` are set

## Setup

### 1. Install Langfuse SDK

```powershell
pip install langfuse
```

### 2. Add Langfuse keys to `.env`

Get a free account at [cloud.langfuse.com](https://cloud.langfuse.com), create a project,
copy the public + secret keys, and add to `.env`:

```env
DIAGNOSTICS_ENABLED=1

LANGFUSE_PUBLIC_KEY=pk-lf-xxxxxxxx
LANGFUSE_SECRET_KEY=sk-lf-xxxxxxxx
LANGFUSE_HOST=https://cloud.langfuse.com

# Optional: also persist full prompts + responses to JSONL/Langfuse.
# Off by default to keep traces small. Turn on for diagnosis sessions.
# DIAGNOSTICS_FULL_PAYLOAD=1
```

### 3. Run the app normally

```powershell
streamlit run app.py
```

The diagnostics hook in `agents/llm_client.py` activates only when
`DIAGNOSTICS_ENABLED=1`. With the flag off, this folder is never imported.

You'll see one line on first LLM activity:

```
✅ diagnostics: instrumentation active (run_id=abc123def456, run_dir=…/runs/2026-04-29_22-15-03)
```

### 4. Analyze the run

After your batch finishes (or any time during it):

```powershell
python -m diagnostics.analyzer
```

This loads the latest run's `traces.jsonl`, generates `summary.md` next to it, and prints
the markdown to stdout. To target a specific run:

```powershell
python -m diagnostics.analyzer diagnostics/runs/2026-04-29_22-15-03
```

JSON output for piping to other tools:

```powershell
python -m diagnostics.analyzer --json > analysis.json
```

## What the summary report contains

The 6 sections map 1:1 to Saumya's diagnostic objectives:

1. **Token breakdown by agent** — which agent burns the most tokens?
2. **TPM rolling windows (60s)** — flagged red where Gemini calls exceed 5 RPM
3. **Per-job aggregates** — does later jobs burn more tokens (state-not-resetting)?
4. **Truncation incidents** — every truncation with prompt size + max_tokens
5. **Burst fan-out** — concurrent-call spikes that cause RPM breach
6. **Auto-derived suspicions** — top-level flags surfaced from the data

## Architecture

```
agents/llm_client.py        ← +5-line conditional hook (deletable)
        │
        ▼ (only if DIAGNOSTICS_ENABLED=1)
diagnostics/
├── __init__.py             ← package version
├── telemetry.py            ← record_llm_call() + ContextVars + Langfuse client
├── instrumentation.py      ← monkey-patches _call_groq, _call_gemini, track_llm_call
├── analyzer.py             ← post-run: traces.jsonl → summary.md
└── runs/<timestamp>/
    ├── manifest.json
    ├── traces.jsonl
    └── summary.md          (generated by analyzer)
```

The instrumentation works by:

- **Wrapping `track_llm_call(agent=...)`** — sets a `current_agent` ContextVar so the
  patched LLM callers know which agent triggered them.
- **Wrapping `_call_groq` and `_call_gemini`** — measures duration, reads the existing
  `_TOKENS_USED_SESSION` counter delta to attribute tokens to this call, detects
  truncation, calls `record_llm_call(...)`.
- **Per-job grouping** — when the orchestrator sets `current_job_id` ContextVar, all
  calls within that scope are tagged so the analyzer can group them.

No agent module is modified. No code paths in `agents/` change behaviour when
diagnostics is disabled.

## Removal

When you're done with diagnosis and want to drop the instrumentation:

```powershell
# 1. Delete this folder
Remove-Item -Recurse -Force diagnostics/

# 2. Edit agents/llm_client.py — remove the "Diagnostics hook" block at the end

# 3. Optional: remove `langfuse` from requirements.txt

# 4. Remove DIAGNOSTICS_*, LANGFUSE_* env vars from .env
```

The main app keeps working with zero behaviour change.

## Limitations

- **Token attribution under concurrency** is approximate. We read the existing
  process-wide `_TOKENS_USED_SESSION` counter delta before/after each call.
  When two threads call simultaneously, each call may attribute the other's
  tokens. Aggregate per-batch totals are correct; per-call tokens may drift.
  For exact attribution we'd need to monkey-patch the SDK clients themselves.
- **Prompt vs completion token split** is unavailable at this layer. We only
  capture the `total_tokens` delta. If you need the split, query Langfuse
  for the SDK-level `usage` object (which it captures separately when you
  use its native integrations).
- **Truncation is heuristic.** Two signals: response length near max_tokens,
  or unclosed JSON. False positives are acceptable; this is a triage flag,
  not ground truth.
